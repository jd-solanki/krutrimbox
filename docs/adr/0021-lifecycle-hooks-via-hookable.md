# Lifecycle hooks via hookable

krutrimbox exposes named lifecycle **hooks** that a repository attaches **Hook Actions** to under a `hooks` key in `.krutrimbox/config.json`, keyed by hook name. The hook bus is [`hookable`](https://github.com/unjs/hookable): configured actions are registered as handlers and fired with `callHook`, which runs them serially and rejects on the first failure — exactly the fail-fast behavior we want. The first hook is `pull-request:ready`, which replaces what would otherwise be a special-cased "review" step: once a Target Issue finishes (every Implementation Issue Resolved), krutrimbox marks the Target Issue Pull Request ready — the only built-in behavior — and then fires `pull-request:ready` against it. With no actions configured, krutrimbox only marks the pull request ready. We chose a general hooks system over a single review feature so future lifecycle points slot in as new hook names without new config shapes, and a small generic action vocabulary over a large map of single-purpose actions.

## Hook actions

- **Agent Action** — runs a fresh Sandboxed Agent session in the Target Issue Sandbox with an operator-supplied prompt (a Markdown file under `.krutrimbox/`, resolved with the existing repository-owned path safety). Its text result is captured as an Action Output (`{{steps.<id>.output}}`). After the session, the host inspects the sandbox workspace; if files changed, the host commits and pushes them (the sandbox keeps Read-Only GitHub Access, so the write happens host-side). The commit carries no `Refs #` footer so it never enters the Done Set.
- **Comment Action** — posts a comment on the pull request. No Factory Comment Marker: a hook runs at most once, so comments are created plainly.
- **Command Action** — runs one allowlisted `gh` command on the host with the Operator's credential, given as an explicit argument array (`run[0]` must be `gh`) and spawned directly — never through a shell, no pipes, each element interpolated individually. Only a curated set of `gh` verb pairs is permitted: `pr ready`, `pr edit`, `pr comment`, `pr review`, `issue comment`, `issue edit`, `label create`. Everything else (including `gh api`, `gh secret`, `pr merge`, `pr close`, `issue delete`, `label delete`) is rejected at config load.

## Lifecycle

Marking the pull request ready is the run-once guard for `pull-request:ready`: it flips draft → ready, so later Factory Runs and Batch re-discovery of a completed-but-unmerged Target Issue see a ready pull request and skip. A failing action aborts the hook (fail-fast) with a diagnostic naming the hook and action; because the pull request is already ready, the operator resolves and re-triggers manually. Actions may interpolate a documented variable surface (`{{pr_number}}`, `{{pr_url}}`, `{{target_issue_number}}`, `{{target_issue_title}}`, `{{target_issue_author}}`, `{{target_issue_branch}}`, `{{operator}}`, `{{base_branch}}`) plus `{{steps.<id>.output}}`.

## Consequences

- **Refines ADR-0013.** That ADR kept prompts non-overridable so krutrimbox owned Sandboxed Agent safety boundaries. Agent Actions deliberately accept fully operator-authored prompts. The boundary is preserved differently: the prompt runs in a Read-Only GitHub sandbox, every GitHub write still happens host-side, Command Actions are gh-verb-allowlisted with no shell, and host auto-commits reference the originating action and prompt for auditability. The committed-config trust rests on ADR-0013's "Project Configuration is reviewed and versioned."
- **Removes ADR-0004.** Automatic routing of final review to the Target Issue Author (and the self-review tag) is gone; reviewer routing is now an operator-authored action (e.g. a `gh pr edit --add-reviewer` Command Action). The conditional self-vs-other logic is no longer provided by krutrimbox.
- **No default behavior beyond mark-ready.** This repository reconstructs its previous built-in review (agent review + comment) as actions on its own committed `pull-request:ready` hook, dogfooding the feature.
- **New runtime dependency: `hookable`.** A small, focused unjs library; the cost is justified by making the hook surface extensible (typed hooks, serial/parallel callers, a future plugin surface) rather than hand-rolling a registry.
