# Use committed krutrimbox project configuration

krutrimbox uses `.krutrimbox/config.json` as the repository-owned Project Configuration file, with configurable Template Slots pointing to Markdown files under `.krutrimbox/`. The directory is committed so teams can review and share krutrimbox policy, while generated runtime state stays local under ignored `.krutrimbox/logs/` and `.krutrimbox/locks/`.

Built-in prompts and templates are stored as Markdown files shipped with the CLI package instead of escaped TypeScript string constants. User configuration may partially override templates by friendly Template Slot names. Prompts stay non-overridable so krutrimbox keeps ownership of Sandboxed Agent safety boundaries, but a repository may inject extra instructions through per-prompt Prompt Extensions: Markdown files referenced under the `prompts` key in `.krutrimbox/config.json` and appended at the tail of the built-in prompt inside a `<repository_instructions>` XML tag. Extensions can only add instructions, never replace krutrimbox's own, and the built-in prompt states that its Non-Negotiable Boundaries take precedence. This keeps krutrimbox generic and skill-agnostic — its own skill invocations and policy (such as requiring JSDoc/docstrings) move out of the shipped prompts into its committed `.krutrimbox/` config — while preserving the safety boundary. krutrimbox injects Factory Comment Markers outside user templates and fails fast on unsupported or invalid project configuration.
